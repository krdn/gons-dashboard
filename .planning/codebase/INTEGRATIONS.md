# External Integrations

**Analysis Date:** 2026-05-11

## APIs & External Services

**LLM (classification + summarization):**
- **Anthropic API via Claude Code CLI Proxy** — server-side LLM calls for email classification.
  - Purpose:
    - `classifyWithLLM` — judges whether an email needs a reply (`needs_reply`, `severity`, `reason`). Code: `src/shared/lib/llm/classify-thread.ts`.
    - `classifyImportantWithLlm` — 4-category importance (money/security/schedule/notice) + summary + rationale. Code: `src/shared/lib/llm/classify-important.ts`.
  - SDK/Client: `@anthropic-ai/sdk` `^0.36.3`, instantiated in `src/shared/lib/llm/anthropic.ts` with `baseURL` + `apiKey` from env. Model constant `HAIKU_MODEL = "claude-haiku-4-5-20251001"`.
  - Endpoint: not Anthropic's public API — points at the user's local Claude Code CLI Proxy at `ANTHROPIC_BASE_URL` (default `http://192.168.0.5:8317`). SDK auto-uses `ANTHROPIC_BASE_URL` / `ANTHROPIC_API_KEY` env vars, but the code passes them explicitly.
  - Env vars: `ANTHROPIC_BASE_URL`, `ANTHROPIC_API_KEY`.
  - Failure modes: Zod schema validation on response, JSON extraction tolerant of code-block wrapping; classifier callers degrade gracefully (`{ kind: "llm-unavailable" }`).
  - Callers: `src/entities/email/api/classifyThread.ts`, `src/entities/email/api/classifyImportant.ts` (invoked by `features/gmail-sync`).

**Gmail (Google Workspace API):**
- **Gmail REST API v1** — base URL `https://gmail.googleapis.com/gmail/v1/users/me`.
  - Purpose: inbox polling, message fetch, label modification.
  - Code locations:
    - OAuth token refresh: `src/shared/api/gmail/auth.ts` (`getValidAccessToken`, hits `https://oauth2.googleapis.com/token`).
    - Incremental polling: `src/shared/api/gmail/history.ts` (`history.list?historyTypes=messageAdded` + `users/me/profile`).
    - Message fetch: `src/shared/api/gmail/messages.ts` (`messages.list`, `messages.get?format=metadata` with `metadataHeaders=From,To,Subject,Date,Reply-To`).
    - Mailing-list signal extraction: `src/shared/api/gmail/headers.ts`.
    - Label modify: `src/shared/api/gmail/modify.ts` (`threads/{id}/modify`, used for mark-as-read / archive).
    - Error classification: `src/shared/api/gmail/errors.ts` — `InvalidGrantError`, `HistoryStaleError`, `GmailRateLimitError`, `GmailServerError`, `GmailClientError`.
    - Public barrel: `src/shared/api/gmail/index.ts`.
  - OAuth scopes (configured in `src/shared/lib/auth/index.ts`):
    - `openid`
    - `https://www.googleapis.com/auth/userinfo.email`
    - `https://www.googleapis.com/auth/gmail.readonly`
  - Auth model: refresh token stored encrypted-at-rest (planned via `PG_ENCRYPTION_KEY` pgcrypto) in the `accounts` table; `getValidAccessToken` auto-refreshes within a 60s clock-skew margin and persists the new `access_token` / `expires_at` back to `accounts`.
  - Retries: `messages.list/get`, `history.list`, `threads.modify` all retry on rate-limit (429) + 5xx with exponential backoff (`MAX_RETRIES=3`, base `500ms`).
  - Failure transitions: an `invalid_grant` response on token refresh sets `users.oauth_state='reauth_required'` + `users.token_expired_at=now()` so polling stops and the UI/email banner triggers (`src/shared/api/gmail/auth.ts`).
  - Env vars: `GOOGLE_CLIENT_ID`, `GOOGLE_CLIENT_SECRET` (used for refresh-token grant); access tokens come from the per-user DB row.

## Data Storage

**Databases:**
- **PostgreSQL 16** — primary store. All persistence goes through Drizzle.
  - Connection: `DATABASE_URL` (default `.env.example` value targets `192.168.0.5:5440`, db `gons_dashboard`).
  - Client: `postgres` driver (`^3.4.5`) wrapped by `drizzle-orm/postgres-js` in `src/shared/lib/db/client.ts` (singleton via `globalThis`, `max: 10`, `idle_timeout: 20`, `connect_timeout: 10`).
  - Schema: `src/shared/lib/db/schema.ts` — Auth.js tables (`users`, `accounts`, `sessions`, `verification_tokens`), email tables (`email_threads`, `reply_needed`, `important_emails`), `push_subscriptions`, server-monitor tables (`hosts`, `projects`, `audit_logs`).
  - Migrations: `drizzle/` directory, generated by `drizzle-kit` (`drizzle.config.ts`). `pnpm db:generate` / `pnpm db:migrate`.
  - At-rest encryption: `pgcrypto` keyed by `PG_ENCRYPTION_KEY` for OAuth refresh tokens (planned column-level wrap).
  - DB access patterns: every read/write goes through `entities/<domain>/api/*` modules marked `import "server-only"`. Examples:
    - `src/entities/email/api/getReplyNeeded.ts` (KST date arithmetic via `NOW() AT TIME ZONE 'Asia/Seoul'`).
    - `src/entities/email/api/getImportantEmails.ts`, `src/entities/email/api/classifyImportant.ts`, `src/entities/email/api/classifyThread.ts`.
    - `src/entities/host/api/getHosts.ts`, `src/entities/host/api/getHostByName.ts`.
    - `src/entities/project/api/getProjects.ts`, `src/entities/project/api/getProjectComposeKeys.ts`, `src/entities/project/api/upsertProjectFromContainer.ts`.
    - `src/features/container-actions/api/insertAuditLog.ts`, `src/features/container-actions/api/_runAction.ts`.

- **Redis 7** — declared as infrastructure (compose service `redis:7-alpine`, port `6390:6379`, AOF on); `REDIS_URL` is required and validated by `src/shared/config/env.ts`, but no Redis client is imported in `src/shared/lib/**` as of this analysis. Reserved for caching / rate-limiting per `CLAUDE.md`.

**File Storage:**
- None. No object-store or local-FS persistence beyond the Postgres volume `gons_dashboard_pgdata` and Redis volume `gons_dashboard_redisdata` (`docker-compose.yml`).

**Caching:**
- Next.js per-route caching only (`export const dynamic = "force-dynamic"` in cron/admin/health routes; `revalidatePath("/servers/<name>")` after container actions in `src/features/container-actions/api/_runAction.ts`).
- Redis container provisioned but not wired to application code yet.

## Authentication & Identity

**Auth Provider:**
- **Auth.js v5 (NextAuth `^5.0.0-beta.25`) with Google OAuth**.
  - Configured in `src/shared/lib/auth/index.ts`.
  - Adapter: `@auth/drizzle-adapter` `^1.7.4` backed by `users`, `accounts`, `sessions`, `verification_tokens` (defined in `src/shared/lib/db/schema.ts`).
  - Session strategy: `database` (cookies hold a session token; user/session rows live in Postgres).
  - Provider: `Google` — `access_type: offline`, `prompt: consent` (forces refresh-token issuance). External-Test OAuth client → refresh tokens expire every 7 days; expiry handled via `users.oauth_state` + `users.token_expired_at` + ops email.
  - `signIn` callback enforces email allowlist (`ALLOWLIST_EMAILS`).
  - On successful re-sign-in, an `events.signIn` resets `users.oauth_state = "active"` and clears `tokenExpiredAt`.
  - Route handler: `src/app/api/auth/[...nextauth]/route.ts`.
  - Custom pages: `/login` (login + error), defined in `src/app/login/page.tsx`.

**Authorization (admin actions):**
- Comma-separated `ADMIN_EMAILS` env list is the allowlist for container actions (`restart`/`start`/`stop`).
- Check helper: `src/features/container-actions/lib/isAdmin.ts`.
- Gate in `src/features/container-actions/api/_runAction.ts` (returns `FORBIDDEN` for non-admins, also enforces Zod input shape — `containerId` must match `/^[a-f0-9]{12,64}$/` to block path traversal).

## Cron & Scheduled Jobs

**Scheduler:**
- Separate cron container running `cron/scheduler.js` (`node-cron` ^3.0.3).
- Code: `cron/scheduler.js`, image `ghcr.io/krdn/gons-dashboard-cron`.
- Timezone: `Asia/Seoul` (passed both via env `TZ` and `node-cron` `timezone: "Asia/Seoul"` — explicit to avoid the UTC-default trap that would fire 8 AM digest at 17:00 KST).

**Jobs:**
| Cron expression | Endpoint called | Purpose | Code |
|-----------------|-----------------|---------|------|
| `0 * * * *` (hourly) | `POST http://app:3020/api/cron/poll-gmail` | Poll Gmail history per active user → classify → upsert `reply_needed` / `important_emails`. | `src/app/api/cron/poll-gmail/route.ts`, drives `syncInbox` from `src/features/gmail-sync/api/syncInbox.ts`. |
| `0 8 * * *` KST (daily) | `POST http://app:3020/api/cron/morning-digest` | Send top-5 reply-needed items as a web-push digest to each active user's subscriptions. | `src/app/api/cron/morning-digest/route.ts`. |
| Startup +30s | `POST http://app:3020/api/cron/poll-gmail` | Catchup poll on container restart. | `cron/scheduler.js`. |

**Auth between cron and app:**
- Bearer token from env `CRON_BEARER_TOKEN`; verified with `timingSafeEqual` in `src/shared/lib/auth/cron.ts` (`verifyCronBearer`). Missing/invalid → HTTP 401.
- Reused by the admin reclassify endpoint `src/app/api/admin/reclassify/route.ts` for manual re-runs via `curl`.

**Incoming:**

- Web Push subscription registration: `POST /api/push/subscribe` (`src/app/api/push/subscribe/route.ts`); deregistration: `DELETE /api/push/subscribe`. Validates payload with Zod and stores in `push_subscriptions`.
- Cron-triggered endpoints: `POST /api/cron/poll-gmail`, `POST /api/cron/morning-digest`, `POST /api/admin/reclassify` (bearer-protected; not webhooks per se, but external HTTP triggers).

**Outgoing:**
- Web Push (see "Push Notifications" below).
- Gmail API calls (see "Gmail").
- Anthropic-compatible LLM proxy calls (see "LLM").
- Google OAuth token endpoint (`https://oauth2.googleapis.com/token`) via `src/shared/api/gmail/auth.ts`.

## Push Notifications

**Web Push (VAPID):**
- Purpose: morning digest delivery for reply-needed emails (top 5/day per user).
- Library: `web-push` `^3.6.7`.
- Code: `src/shared/lib/push/index.ts` (`sendPush`, lazy VAPID configuration; returns discriminated union `sent | vapid-missing | expired | error`).
- TTL: `6 * 60 * 60` seconds (6h).
- Expired endpoints (HTTP 404/410) are reported back so callers can delete the row — the morning digest route does this inline (`src/app/api/cron/morning-digest/route.ts`).
- Subscription storage: `push_subscriptions` table (`endpoint` unique, `p256dh`, `auth`, `user_id`).
- Subscription registration handler: `src/app/api/push/subscribe/route.ts` — requires authenticated session (`auth()` from NextAuth).
- Client-side trigger: `src/widgets/email-digest/ui/PushSubscribeButton.tsx`.
- Env vars: `VAPID_PUBLIC_KEY`, `VAPID_PRIVATE_KEY`, `VAPID_SUBJECT` (optional — if any is missing, `sendPush` returns `{ kind: "vapid-missing" }` and no notifications go out).

## Docker daemon integration (server monitor)

**Docker CLI over context:**
- Purpose: list/inspect/restart containers on registered hosts as part of the Server Infra Monitor domain.
- Mechanism: `docker --context <ctx> <args>` invoked via `execFile` (never via shell — argument array only). Transport (local socket vs. SSH) is owned by the docker CLI's context config, not the app.
- Code:
  - `src/shared/lib/docker/runDocker.ts` — `execFile("docker", ["--context", ctx, ...args])` with timeout from `DOCKER_CMD_TIMEOUT_MS` (default 10s) and `maxBuffer: 4MB`.
  - `src/shared/lib/docker/listContainers.ts` — `docker container ls --all --no-trunc --format "{{json .}}"`.
  - `src/shared/lib/docker/inspectContainer.ts` — `docker inspect <id>`; output validated with Zod (`InspectShape`), env vars masked by `src/shared/lib/docker/maskEnv.ts`.
  - `src/shared/lib/docker/parseContainer.ts` — JSON-per-line parser.
  - Index/public barrel: `src/shared/lib/docker/index.ts`.
- Higher-level wrappers: `src/entities/container/api/listContainers.ts`, `src/entities/container/api/inspectContainer.ts`.
- Mutating actions (`restart`, `start`, `stop`) all funnel through `src/features/container-actions/api/_runAction.ts`, which enforces:
  1. NextAuth session present (`UNAUTHORIZED` otherwise),
  2. Email in `ADMIN_EMAILS` (`FORBIDDEN` otherwise),
  3. Zod input (`hostId` uuid, `containerId` `/^[a-f0-9]{12,64}$/`, `containerName` 1–200 chars) (`INVALID_INPUT` otherwise),
  4. Host exists in DB (`HOST_NOT_FOUND` otherwise),
  5. Action runs via `runDocker`; result + duration recorded in `audit_logs` (status `success` / `failed`).
- Host registry: `hosts` table (`docker_context` column carries the CLI context name); `pnpm db:seed:hosts` populates initial rows (`home-server`, `krdn-lenovo` per `CLAUDE.md`).
- Env vars: `DOCKER_DEFAULT_CONTEXT` (validation hint at startup), `DOCKER_CMD_TIMEOUT_MS`, `ADMIN_EMAILS`.

**Optional env vars:**
- `VAPID_PUBLIC_KEY`, `VAPID_PRIVATE_KEY`, `VAPID_SUBJECT` (must start with `mailto:`) — push notifications.
- `OPS_NOTIFY_EMAIL` — ops alert delivery.
- `PG_ENCRYPTION_KEY` (≥32) — pgcrypto wrap for refresh tokens at rest.

**Secrets location:**
- `.env` on the deployment host (gitignored). `Dockerfile` and CI inject only placeholders so Zod validation passes during builds; real secrets are injected at `docker compose up` time via the host's `.env` file referenced by `docker-compose.yml`.
- Per project policy, **no secret values** are ever written into source, docs, commit messages, or markdown — only variable names.
