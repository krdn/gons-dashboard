# 도메인 문서

엔지니어링 스킬이 코드베이스를 탐색할 때 이 저장소의 도메인 문서를 어떻게 읽어야 하는지 정의합니다.

## 탐색 전에 읽을 것

- **`CONTEXT.md`** — 저장소 루트의 도메인 용어집과 핵심 개념
- **`docs/adr/`** — 작업 영역과 관련된 ADR(Architectural Decision Record)을 모두 읽음

이 파일들이 아직 없다면 **조용히 진행합니다**. 누락 사실을 따로 보고하거나 미리 만들자고 제안하지 마세요. 생산자 스킬(`/grill-with-docs`)이 용어나 결정이 실제로 합의될 때 lazy하게 생성합니다.

## 파일 구조

이 저장소는 **단일 컨텍스트** 레이아웃입니다:

```
/
├── CONTEXT.md
├── docs/
│   ├── agents/                  ← 에이전트 설정 (이 폴더)
│   └── adr/
│       ├── 0001-fsd-architecture.md
│       ├── 0002-email-provider-선택.md
│       └── ...
└── src/
    ├── app/
    ├── widgets/
    ├── features/
    ├── entities/
    └── shared/
```

FSD 슬라이스(`features/email-analysis/` 등)에는 자체 README를 둘 수 있지만 별도 `CONTEXT.md`는 두지 않습니다 — 단일 앱 안의 도메인이므로 루트 `CONTEXT.md`로 통합 관리합니다.

## 용어집(glossary)의 어휘를 사용

출력에서 도메인 개념을 명명할 때(이슈 제목, 리팩터링 제안, 가설, 테스트 이름 등), `CONTEXT.md`에 정의된 용어를 그대로 씁니다. 용어집이 명시적으로 피하는 동의어로 표류하지 마세요.

필요한 개념이 아직 용어집에 없다면 두 가지 신호 중 하나입니다:

1. **프로젝트가 쓰지 않는 언어를 만들고 있다** → 다시 생각합니다
2. **실제로 빈 자리가 있다** → `/grill-with-docs`로 채울 항목으로 메모합니다

## ADR 충돌은 명시적으로 표면화

출력이 기존 ADR과 모순되면 조용히 덮어쓰지 말고 명시적으로 드러냅니다:

> _ADR-0007(이벤트 소싱 주문)과 모순됨 — 그래도 다시 검토할 가치가 있는 이유는…_

## 한국어 도메인 용어

이 프로젝트는 한국어로 문서를 작성합니다. 도메인 용어는 다음과 같이 처리합니다:

- **한국어 우선**: 사용자에게 노출되는 개념(예: "중요한 메시지", "답변 필요")은 한국어로 정의
- **기술 용어 병기**: 처음 등장 시 영어 원문 병기 (예: "요약(summary)", "필터링(filter)")
- **코드 식별자**: 영어로만 작성 (`importantMessages`, `requiresReply`)

`CONTEXT.md`에는 한국어 용어와 그에 대응하는 영어 식별자를 함께 기재합니다.
